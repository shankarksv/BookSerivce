Entities Required

1. Book
- id: Long
- title: String
- author: String
- year: Integer
- description: String
- price: BigDecimal

Notes:
- Each book price is fixed at 50 EUR
- title is required
- author is required
- year must be valid


2. Basket
- id: Long
- userId: Long
- status: String
- createdAt: LocalDateTime
- updatedAt: LocalDateTime
- items: List<BasketItem>

Notes:
- One active basket per user is enough for the initial version
- status can start with ACTIVE and CLEARED


3. BasketItem
- id: Long
- basketId: Long
- bookId: Long
- quantity: Integer

Notes:
- quantity must be greater than 0
- one basket item per distinct book inside a basket


Relationships

- One basket belongs to one user
- One basket has many basket items
- One basket item references one book


Book Inventory Controller

Controller Name:
- BookController

Base Path:
- /api/books

Endpoints:

1. GET /api/books
- Purpose: get all books in inventory
- Response: list of books

2. GET /api/books/{id}
- Purpose: get one book by id
- Response: single book

3. POST /api/books
- Purpose: create a new book
- Request Body:
  - title
  - author
  - year
  - description
- Response: created book
- Status: 201 Created

4. PUT /api/books/{id}
- Purpose: update book details
- Request Body:
  - title
  - author
  - year
  - description
- Response: updated book
- Status: 200 OK

5. DELETE /api/books/{id}
- Purpose: delete a book from inventory
- Rule: do not delete if the book is present in an active basket
- Status: 204 No Content

Basic Authentication

Goal:
- secure APIs using Spring Security basic authentication for the initial version

Initial approach:
- Use Spring Security with in-memory users for development/testing
- Send username and password in Authorization header using Basic Auth
- Passwords should be encoded using BCryptPasswordEncoder

Roles:
- ADMIN
  - access to create, update, and delete books
- USER
  - access to read books and manage basket

Basic auth design:
- SecurityConfig class required
- Configure authentication provider for in-memory users
- Configure endpoint authorization rules
- Disable CSRF for stateless REST API if needed

Example access rules:
- GET /api/books -> ADMIN, USER
- POST /api/books -> ADMIN
- PUT /api/books/{id} -> ADMIN
- DELETE /api/books/{id} -> ADMIN

Variable Discount Pricing Refactor

Current pricing design
- Pricing now uses a configurable discount table by group size
- Default discount table:
  - 2 books -> 5 percent
  - 3 books -> 10 percent
  - 4 books -> 20 percent
  - 5 books -> 25 percent
- PricingService no longer uses a hardcoded 320 EUR exception
- BasketItemLine is still the pricing input model for unit testing and algorithm work

Optimal pricing approach
- For small and medium baskets, pricing uses a memoized dynamic programming solver
- The solver normalizes remaining quantities into a sorted quantity state
- The solver tries all valid distinct-book groupings and keeps the minimum total
- This makes the engine choose the best customer discount for tested grouping combinations
- For very large baskets, a layered fallback is still used to keep runtime practical

Persistence hooks for future work
- DiscountConfig added as a pricing domain shape
- DiscountPolicyProvider added as the contract for loading active discounts
- InMemoryDiscountPolicyProvider added as the first implementation
- This keeps pricing logic plain Java for now, while giving us a clean seam for future DB-backed discount config

Variable discount test coverage added
- same basket priced with default and custom discount tables
- missing discount table entry falls back to zero discount for that group size
- stronger 4-book discount changes total as expected
- JSON pricing cases can now optionally pass a discount table
